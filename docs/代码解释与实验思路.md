# 代码解释与实验思路

## 一、矩阵每列与向量内积
### 1. `dot_naive`
```c
for (i = 0; i < n; ++i)
    for (j = 0; j < n; ++j)
        sum[i] += b[j*n + i] * a[j];
```
含义：一次求出一列的内积。问题在于 `b[j*n + i]` 会隔着整行跳着访问矩阵，空间局部性差。

### 2. `dot_cache`
```c
for (j = 0; j < n; ++j)
    for (i = 0; i < n; ++i)
        sum[i] += row[i] * a[j];
```
含义：改成按行顺序访问矩阵。虽然总乘加次数没变，但内层访问连续，更容易命中 cache。

### 3. `dot_cache_unroll4`
在 `dot_cache` 的基础上，每轮循环同时处理 4 个元素，减少循环判断和归纳变量更新的开销。这属于进阶要求中的循环展开。

## 二、n 个数求和
### 1. `sum_naive`
```c
sum += a[i];
```
含义：所有加法都挂在同一条依赖链上，后一条加法必须等待前一条完成。

### 2. `sum_dual`
```c
s0 += a[i];
s1 += a[i+1];
```
含义：把一条长依赖链拆成两条独立链，处理器更容易把它们并行发射到不同流水线。

### 3. `sum_dual_unroll4`
把两条链进一步扩展到四条链，同时降低循环本身的簿记开销。这既体现了超标量，也体现了循环展开。

### 4. `sum_pairwise`
两两相加、逐层归约。理论上并行结构更好，但当前串行实现需要额外写回和多轮遍历，所以在你的机器上可能并不占优。这个结果反而很适合写分析：算法“更像并行”不代表串行 CPU 上一定更快。

## 三、计时方法为什么要重复运行
单次运行可能只有微秒级，计时误差会很大。因此 `main.c` 会自动根据问题规模估计重复次数，并保证一次测量至少持续约 0.2 秒，再取平均时间。

## 四、为什么程序先做正确性检查
性能对比必须建立在结果正确的前提下。程序在正式测试前会让所有版本在同一组固定数据上运行，并检查输出是否一致；如果不一致就直接终止。
